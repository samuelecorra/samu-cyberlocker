## 1. Panoramica della lezione

Questa lezione si compone di due parti. La prima riprende brevemente gli **esercizi OpenJML** della lezione precedente (Sokoban, esempi con metodi statici, classi con invarianti). La seconda — il cuore della lezione — introduce gli **assert di Java**, un meccanismo nativo del linguaggio per esprimere proprietà di correttezza direttamente nel codice sorgente, senza ricorrere a strumenti esterni come JML. [p. 1–2]

Gli assert rappresentano l'incarnazione più semplice del principio di **Design by Contract** in Java: il programmatore dichiara formalmente le proprie ipotesi sul comportamento del programma e il runtime le verifica automaticamente.

## 2. Che cos'è un assert

Un **assert** è un'istruzione Java che contiene un'espressione booleana la quale si ritiene essere **vera** nel punto in cui viene eseguita. Se l'espressione risulta falsa, il runtime lancia un `AssertionError` (un'eccezione non controllata), segnalando la presenza di un **bug** nel programma. [p. 3]

### 2.1 Proprietà fondamentali degli assert

- Un assert è una **dichiarazione di una formula booleana** che il programmatore _pensa_ debba valere in un certo punto del programma.
- Un assert **non dovrebbe modificare la logica** del programma: il comportamento del programma deve essere identico con o senza assert abilitati.
- Se un assert è falso, **c'è un bug nel programma** (non nell'input dell'utente).

```java
d = a * a;
assert d >= 0;  // Il quadrato di un numero è sempre >= 0
```

> [!IMPORTANT]
> Gli assert servono a rilevare **errori di programmazione** (situazioni che _non dovrebbero_ mai verificarsi), non condizioni anomale dell'ambiente che _possono_ verificarsi (per quelle si usano le eccezioni). Questa distinzione è fondamentale nella progettazione di software sicuro.

## 3. Sintassi degli assert in Java

Java offre due forme sintattiche per gli assert. [p. 4–5]

### 3.1 Forma semplice

```java
assert Espressione1;
```

- `Espressione1` è un'espressione booleana.
- Se `Espressione1` è `false`, viene lanciato un `AssertionError` **senza** messaggio dettagliato.

### 3.2 Forma con messaggio diagnostico

```java
assert Espressione1 : Espressione2;
```

- `Espressione1` è un'espressione booleana.
- `Espressione2` è un'espressione con un valore (tipicamente una stringa o un valore numerico).
- Se `Espressione1` è `false`, viene lanciato un `AssertionError` il cui messaggio contiene la rappresentazione in stringa di `Espressione2`.

```java
assert counter >= 0 : "counter = " + counter;
// Se counter è -3, il messaggio sarà: "counter = -3"
```

> [!TIP]
> La forma con messaggio è **fortemente consigliata** in pratica: quando un assert fallisce in produzione o durante il testing, il messaggio consente di diagnosticare immediatamente il problema senza dover riprodurre lo scenario nel debugger.

## 4. Assert e side effect

Le espressioni contenute in un assert **devono essere prive di effetti collaterali** (_side effect free_): la valutazione dell'espressione non deve influire su alcuno stato visibile del programma. [p. 5]

Il motivo è semplice: gli assert possono essere **abilitati o disabilitati** a runtime (tramite i flag `-ea` / `-da` della JVM). Se un assert avesse effetti collaterali, il comportamento del programma cambierebbe a seconda che gli assert siano attivi o meno — una situazione inaccettabile e fonte di bug estremamente insidiosi.

### 4.1 Esempio di violazione

```java
// SBAGLIATO: persona.remove() ha un side effect!
assert persona.remove();
// Con assert abilitati: rimuove la persona e verifica il risultato
// Con assert disabilitati: NON rimuove la persona → bug silenzioso
```

**Soluzione corretta:**

```java
// L'effetto collaterale è separato dall'assert
boolean removed = persona.remove();
assert removed;
// Il programma funziona correttamente sia con che senza assert
```

> [!WARNING]
> Questa regola è l'esatto parallelo del concetto di **metodo puro** in JML: solo espressioni pure (senza effetti collaterali) possono apparire nelle specifiche. La violazione di questo principio è una delle fonti più comuni di bug legati alla sicurezza, perché crea discrepanze tra il comportamento in ambiente di sviluppo (assert abilitati) e produzione (assert disabilitati).

## 5. Pattern d'uso degli assert

### 5.1 Verifica di invarianti impliciti nel flusso di controllo

Quando una catena di `if-else` copre tutti i casi possibili, l'ultimo `else` può contenere un assert che documenta e verifica l'assunzione del programmatore: [p. 6]

**Senza assert (assunzione implicita):**

```java
if (i % 3 == 0) {
    // ...
} else if (i % 3 == 1) {
    // ...
} else {
    // Caso in cui i % 3 == 2 (assunzione implicita)
    // ...
}
```

**Con assert (assunzione esplicita e verificata):**

```java
if (i % 3 == 0) {
    // ...
} else if (i % 3 == 1) {
    // ...
} else {
    assert i % 3 == 2 : i;  // Rende esplicita e verificabile l'assunzione
    // ...
}
```

### 5.2 Punti di codice irraggiungibili

Quando il flusso di esecuzione non dovrebbe _mai_ raggiungere un certo punto del programma, `assert false` lo dichiara esplicitamente: [p. 6]

**Senza assert:**

```java
void foo() {
    for (/* ... */) {
        if (/* ... */)
            return;
    }
    // L'esecuzione non dovrebbe mai arrivare qui
}
```

**Con assert:**

```java
void foo() {
    for (/* ... */) {
        if (/* ... */)
            return;
    }
    assert false;  // Se arriviamo qui, c'è un bug
}
```

> [!NOTE]
> `assert false` è un idioma standard in Java per marcare codice morto (_dead code_). È particolarmente utile nei blocchi `default` di `switch exhaustive`, dove tutti i casi sono già coperti ma il compilatore richiede comunque un'azione per il `default`.

## 6. Assert e metodi pubblici vs non pubblici

La convenzione Java per l'uso degli assert nelle **precondizioni** segue una regola chiara che riflette il principio del Design by Contract. [p. 7]

### 6.1 Precondizioni su metodi `public`: NO assert

Per i metodi pubblici, le precondizioni vengono implementate con **controlli espliciti** che lanciano eccezioni specifiche (tipicamente `IllegalArgumentException`, `NullPointerException`, `IllegalStateException`): [p. 7]

```java
public void setRefreshRate(int rate) {
    // Precondizione: controllo esplicito con eccezione
    if (rate <= 0 || rate > MAX_REFRESH_RATE)
        throw new IllegalArgumentException("Illegal rate: " + rate);

    setRefreshInterval(1000 / rate);
}
```

**Perché non si usano assert?** Perché:

1. Gli assert possono essere **disabilitati** a runtime, rendendo il controllo non affidabile.
2. I metodi pubblici fanno parte dell'**interfaccia contrattuale** della classe: la violazione di una precondizione pubblica è responsabilità del **chiamante**, e deve essere gestita con un'eccezione che il chiamante può intercettare.

### 6.2 Precondizioni su metodi non `public`: SÌ assert

Per i metodi privati (o package-private, o protected), le precondizioni sono **interne** alla classe o al package. Le violazioni indicano errori del programmatore della stessa unità di codice, ed è appropriato usare gli assert: [p. 7]

```java
/**
 * Sets the refresh interval (must correspond to a legal frame rate).
 * @param interval refresh interval in milliseconds.
 */
private void setRefreshInterval(int interval) {
    // Precondizione su metodo non pubblico: assert appropriato
    assert interval > 0 && interval <= 1000 / MAX_REFRESH_RATE
        : "interval = " + interval;
    // ...
}
```

> [!IMPORTANT]
> **Regola pratica:**
>
> | Tipo di metodo                    | Precondizioni con…            | Motivo                                     |
> | --------------------------------- | ----------------------------- | ------------------------------------------ |
> | `public`                          | Eccezioni esplicite (`throw`) | Contratto verso l'esterno, sempre attivo   |
> | `private` / `protected` / package | `assert`                      | Verifica interna, errore del programmatore |
>
> Questa distinzione è fondamentale per la sicurezza: un metodo pubblico di un'API esposta non può fare affidamento sugli assert per validare l'input.

## 7. Assert nelle postcondizioni

Gli assert possono essere usati per verificare **postcondizioni** sia nei metodi pubblici sia in quelli non pubblici. [p. 8]

### 7.1 Esempio: verifica del valore di ritorno

```java
public static int getTrueIndex(boolean[] arrBool) {
    for (int i = 0; i < arrBool.length; i++) {
        if (arrBool[i]) {
            assert arrBool[i];   // Postcondizione: l'elemento trovato è true
            return i;
        }
    }
    // Se arriviamo qui, nessun elemento è true
    for (int i = 0; i < arrBool.length; i++) {
        assert !arrBool[i];      // Postcondizione: tutti gli elementi sono false
    }
    return -1;
}
```

### 7.2 Il problema del valore precedente (`\old`)

In JML, `\old(expr)` consente di riferirsi al valore di un'espressione _prima_ dell'esecuzione del metodo. In Java puro, questa funzionalità non esiste: il programmatore deve **salvare manualmente** il valore precedente in una variabile locale. [p. 8]

**In JML:**

```java
//@ ensures counter == \old(counter) - 1;
public void decr() {
    counter--;
}
```

**Equivalente con assert Java:**

```java
public void decr() {
    int oldcounter = counter;    // Salvataggio manuale del valore precedente
    counter--;
    assert counter == oldcounter - 1;  // Verifica postcondizione
}
```

> [!TIP]
> Questo pattern — salvare il vecchio stato, eseguire l'operazione, verificare la relazione tra vecchio e nuovo stato — è la traduzione operativa del concetto di postcondizione. È più verboso di JML, ma funziona senza strumenti esterni.

## 8. Assert per gli invarianti di classe

Gli **invarianti di classe** devono valere: [p. 9]

1. **Dopo** ogni costruttore.
2. **Prima** di ogni metodo pubblico.
3. **Dopo** ogni metodo pubblico.

In JML, un invariante viene dichiarato una volta e verificato automaticamente. In Java puro, il programmatore deve inserire manualmente gli assert in tutti i punti rilevanti.

### 8.1 Pattern di implementazione

Si definisce un **metodo privato** che verifica l'invariante, e lo si chiama in tutti i punti necessari:

```java
// Metodo helper per l'invariante
private boolean counterPositivo() {
    return counter > 0;
}

// Ogni costruttore: assert DOPO l'inizializzazione
CounterAssert() {
    counter = 0;
    assert counterPositivo() : "counter = " + counter;
}

// Ogni metodo: assert ALL'INIZIO e ALLA FINE
public void incr() {
    assert counterPositivo() : "counter = " + counter;  // Invariante pre
    // ... corpo del metodo ...
    assert counterPositivo() : "counter = " + counter;  // Invariante post
}
```

## 9. Esempio completo: `CounterAssert.java`

L'esempio seguente combina tutti i concetti in una classe completa: invarianti, precondizioni (implicite), postcondizioni e verifica manuale del valore precedente. [p. 11]

```java
public class CounterAssert {

    private int counter;

    CounterAssert() {
        counter = 0;
        // counter = -1;  // Inizializzazione errata (per test)
        assert counter >= 0 : "counter = " + counter;  // Invariante post-costruttore
    }

    //@ ensures counter == \old(counter) + 1;
    public void incr() {
        assert counter >= 0 : "counter = " + counter;  // Invariante pre-metodo
        int oldcounter = counter;                        // Salvataggio per postcondizione
        counter++;
        assert counter == oldcounter + 1;                // Postcondizione
        // assert counter == oldcounter - 1;             // Assert errato (per test)
        assert counter >= 0 : "counter = " + counter;  // Invariante post-metodo
    }

    //@ ensures counter == \old(counter) - 1;
    public void decr() {
        assert counter >= 0 : "counter = " + counter;  // Invariante pre-metodo
        int oldcounter = counter;                        // Salvataggio per postcondizione
        if (counter > 0)
            counter--;
        assert counter == oldcounter - 1;                // Postcondizione
        assert counter >= 0 : "counter = " + counter;  // Invariante post-metodo
    }

    public static void main(String[] args) {
        CounterAssert c = new CounterAssert();
        c.incr();
        c.decr();
    }
}
```

> [!WARNING]
> Si noti un **bug intenzionale** nel metodo `decr()`: la postcondizione `assert counter == oldcounter - 1` fallisce quando `counter == 0`, perché il `if (counter > 0)` impedisce il decremento ma l'assert si aspetta comunque che il contatore sia diminuito. Questo illustra come gli assert possano **scoprire incoerenze** tra il contratto (postcondizione JML) e l'implementazione.

## 10. Assert Java vs Assert JML

Sebbene entrambi utilizzino la parola chiave `assert`, gli assert JML sono **più espressivi** di quelli Java. [p. 9]

### 10.1 Confronto: verificare che un array non contenga `null`

**Con assert Java:**

```java
// Serve un ciclo esplicito
for (int n = 0; n < a.length; n++)
    assert a[n] != null;
```

**Con assert JML (all'interno del codice):**

```java
/*@ assert (\forall int i; 0 <= i && i < n; a[i] != null); @*/
```

La versione JML è più compatta e dichiarativa grazie al quantificatore `\forall`. In Java puro, il programmatore deve implementare esplicitamente l'iterazione.

| Caratteristica                        | Assert Java               | Assert JML   |
| ------------------------------------- | ------------------------- | ------------ |
| Quantificatori (`\forall`, `\exists`) | No                        | Sì           |
| `\result`, `\old()`                   | No                        | Sì           |
| Implicazione (`==>`)                  | No (richiede `!a \|\| b`) | Sì           |
| Utilizzabili nelle specifiche         | No                        | Sì           |
| Verificabili a runtime                | Sì (con `-ea`)            | Sì (con RAC) |

## 11. Assert vs eccezioni

La distinzione tra assert ed eccezioni è concettuale e non va confusa: [p. 10]

|                     | Assert                                         | Eccezioni                                       |
| ------------------- | ---------------------------------------------- | ----------------------------------------------- |
| **Scopo**           | Rilevare **errori di programmazione**          | Gestire **condizioni anomale**                  |
| **Quando scattano** | Situazioni che **non dovrebbero** mai accadere | Situazioni che **possono** accadere             |
| **Disabilitabili**  | Sì (flag JVM `-da`)                            | No                                              |
| **Esempio**         | Invariante violato, postcondizione falsa       | File non trovato, input utente invalido         |
| **Errore generato** | `AssertionError`                               | `IOException`, `IllegalArgumentException`, ecc. |

> [!IMPORTANT]
> In un sistema software sicuro, le **eccezioni** proteggono dai comportamenti imprevisti dell'ambiente esterno (input malevoli, risorse mancanti), mentre gli **assert** proteggono dagli errori logici interni. L'uso combinato dei due meccanismi fornisce una difesa in profondità (_defense in depth_) contro i bug.

## 12. Abilitare gli assert in Eclipse

Per impostazione predefinita, la JVM **disabilita** gli assert. Per attivarli in Eclipse: [p. 10]

1. **Run → Run Configurations…**
2. Selezionare la configurazione di esecuzione desiderata.
3. Nella scheda **Arguments**, nel campo **VM arguments**, aggiungere:

```
-ea
```

(`-ea` sta per _enable assertions_; `-da` le disabilita.)

![Schermata delle Run Configurations di Eclipse con il flag -ea](link_immagine_eclipse_ea)

> [!TIP]
> In un ambiente di sviluppo, gli assert dovrebbero essere **sempre abilitati**. In produzione, la scelta dipende dalla criticità del sistema: per software dove la sicurezza è la priorità (come quello trattato in questo corso), è buona pratica tenerli abilitati anche in produzione, accettando il modesto costo di performance in cambio della rilevazione precoce di errori.

| ------------------------ | ------------------------------------------------------------------ |
| **Assert**               | Istruzione Java che verifica un'ipotesi del programmatore          |
