## 1. Validazione e Verifica (V&V) [p. 2]

| Concetto                    | Definizione                                                                            | Motto                         |
| --------------------------- | -------------------------------------------------------------------------------------- | ----------------------------- |
| **Validazione** (convalida) | Controlla che il SW sia conforme ai **requisiti dell'utente** (funzionalità richieste) | _"Building the right system"_ |
| **Verifica**                | Controlla che l'implementazione corrisponda alla sua **specifica**                     | _"Building the system right"_ |

> [!info] La verifica richiede un modello dei requisiti
> Senza una specifica formale o semi-formale, non è possibile verificare la correttezza dell'implementazione.

## 2. Tecniche di V&V a livello di implementazione [pp. 3–4]

Le tecniche di V&V dell'implementazione si dividono in due grandi famiglie:

### 2.1 Tecniche dinamiche [p. 4]

Analizzano informazioni sul SW **eseguendolo**:

| Tecnica                       | Descrizione                                                                                                                                                       |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Runtime checkers**          | Analizzano l'esecuzione del codice per rilevare violazioni di memoria, deadlock, ecc. prima di passare le chiamate al SO (es. LibSafe, PurifyPlus, Immunix tools) |
| **Profilers**                 | Valutano le prestazioni ed evidenziano comportamenti anomali (non necessariamente scorretti)                                                                      |
| **Penetration testing tools** | Scansionano l'applicazione secondo schemi di vulnerabilità conosciuti                                                                                             |
| **Testing**                   | Esecuzione di scenari per la ricerca di difetti (random testing o con criteri)                                                                                    |

## 3. Analisi statica in compilazione [pp. 6–7]

I compilatori effettuano una analisi statica del codice per verificare che un programma soddisfi particolari caratteristiche di **correttezza statica**, necessarie per generare il codice oggetto.

Tipiche anomalie identificabili (dipendono dal linguaggio):

- Nomi di **identificatori non dichiarati**
- Incoerenza tra **tipi di dati** coinvolti in un'istruzione
- Incoerenza tra **parametri formali ed effettivi** in chiamate a subroutine
- **Codice non raggiungibile** dal flusso di controllo

## 4. Revisione tra pari (Peer Review) [pp. 8–9]

Tecnica per valutare in modo **anonimo** il codice per qualità, leggibilità e usabilità.

### 4.1 Processo [p. 8]

1. A ogni partecipante viene chiesto di consegnare due programmi: uno _"migliore"_ e uno di qualità inferiore
2. I programmi vengono distribuiti **casualmente** tra i partecipanti
3. Ogni partecipante valuta **4 programmi** (2 migliori + 2 di qualità inferiore)
4. Dopo la raccolta, ogni partecipante riceve le valutazioni **anonime** dei propri programmi e le statistiche

### 4.2 Pratica moderna [p. 9]

È una pratica standard in:

- Aziende di software, soprattutto in **progetti open source**
- **Team agili**
- Sviluppo tramite **GitHub/GitLab**

Essenziale per:

- Migliorare la **qualità del codice** e ridurre i bug
- **Condividere conoscenza** nel team
- Mantenere uno **stile coerente**
- Fare **mentoring** tra sviluppatori

> [!tip] Strumenti moderni
> **GitHub Pull Request** e **GitLab Merge Request** integrano automaticamente il processo di peer review.

## 5. Ispezione — Modello Fagan [pp. 10–16]

### 5.1 Caratteristiche [p. 10]

L'**ispezione di codice secondo il modello Fagan** è tra le tecniche di maggior successo:

- Più **formale e ben strutturata** di altre proposte
- Tecnica completamente **manuale** per trovare e correggere errori
- Estensibile a **tutti gli artefatti** del ciclo di vita del SW (progetto, requisiti, ecc.)

### 5.2 Ruoli [p. 11]

| Ruolo             | Responsabilità                                                                           |
| ----------------- | ---------------------------------------------------------------------------------------- |
| **Moderatore**    | Organizza le riunioni, presiede le sedute, sceglie i partecipanti, controlla il processo |
| **Revisori** (2)  | Leggono il codice, cercano difetti                                                       |
| **Programmatore** | Partecipante passivo; risponde a domande quando richiesto                                |

### 5.3 Checklist di ispezione [pp. 13–14]

Le checklist sono divise per aree:

- **Funzionalità**, uso dei dati, flusso di controllo, connessioni, calcolo, manutenzione, chiarezza del codice

> [!example] Esempi di difetti cercati nelle checklist
>
> - **Uso improprio dei dati:** variabili non inizializzate, indice di array fuori limite, puntatori _dangling_
> - **Errori nelle dichiarazioni:** uso di variabili non dichiarate o dichiarazione omonima in blocchi annidati
> - **Errori di calcolo:** divisione per zero, overflow (anche in risultati intermedi), errori di tipo, priorità operatori
> - **Difetti nelle espressioni relazionali:** operatore errato (`>` invece di `≥`, `=` invece di `==`), priorità operatori booleani
> - **Errori nel flusso di controllo:** loop infiniti, loop eseguito $n+1$ o $n-1$ volte anziché $n$
> - **Errori nelle interfacce:** numero errato di parametri, tipo errato, uso incoerente di variabili globali

> [!info] Domande di approfondimento
> Per evitare la superficialità, i revisori devono rispondere a domande come:
>
> - _"In quali condizioni può essere chiamata questa funzione?"_
> - _"Qual è l'effetto di questa funzione sul comportamento di altre funzioni?"_ (analisi delle dipendenze)

### 5.4 Incentivi [p. 16]

> [!warning] Regola fondamentale
> I difetti trovati **nell'ispezione** non sono usati nella valutazione personale → i programmatori non hanno motivo di nascondere i difetti.
>
> I difetti trovati durante il **test** (dopo l'ispezione) sono usati nella valutazione personale → i programmatori sono incentivati a trovare difetti nell'ispezione.

## 6. Walkthrough [p. 17]

Analisi **informale** del codice in cui i partecipanti _"operano come il computer"_:

- Si scelgono alcuni **casi di test** e si simula l'esecuzione del codice **a mano** (si _attraversa_ — walkthrough — il codice)

| Aspetto          | Dettaglio                                                |
| ---------------- | -------------------------------------------------------- |
| **Partecipanti** | Tra 3 e 5, senza ruoli rigidi                            |
| **Durata**       | Max 120 minuti                                           |
| **Focus**        | Ricerca dei difetti, non correzione                      |
| **Chi guida**    | L'**autore** del codice (differenza chiave vs ispezione) |

> [!tip] Walkthrough vs Ispezione
> La principale differenza è l'**informalità** e il fatto che è l'**autore** (non un moderatore) a guidare la revisione.

## 7. Analisi statica automatica del codice [pp. 18–19]

### 7.1 Modelli di rappresentazione [p. 18]

| Modello                        | Descrizione                                    |
| ------------------------------ | ---------------------------------------------- |

### 7.2 Metriche di qualità del codice [p. 19]

L'analisi statica è usata già dal compilatore per **ottimizzare**, ma valuta anche la **qualità** del codice (non solo la correttezza):

| #   | Metrica                             | Cosa misura                              |
| --- | ----------------------------------- | ---------------------------------------- |
| 1   | **Stile del codice**                | Aderenza a convenzioni                   |
| 2   | **Costrutti sospetti** (Code smell) | Pattern che indicano potenziali problemi |
| 3   | **Codice duplicato**                | Ripetizioni eliminabili                  |
| 4   | **Accoppiamento**                   | Dipendenze tra moduli                    |
| 5   | **Complessità**                     | Complessità ciclomatica e strutturale    |

## 8. Control Flow Analysis [pp. 22–29]

### 8.1 Definizione [p. 22]

Il flusso di controllo è esaminato per verificarne la correttezza. Il codice è rappresentato tramite un **Control Flow Graph (CfG)**:

- **Nodi** = statement (istruzioni e/o predicati) del programma
- **Archi** = passaggio del flusso di controllo

Un CfG è esaminato per:

- Identificare **ramificazioni** del flusso di controllo
- Verificare l'esistenza di eventuali **anomalie** (codice irraggiungibile, non strutturazione)

### 8.2 Vantaggi del CfG [p. 23]

> [!tip] Benefici dell'analisi CfG
>
> - Permette di **visualizzare** come viene eseguito il codice
> - Aiuta a trovare errori: rileva **codice irraggiungibile** o **loop infiniti**
> - Utile per l'**ottimizzazione**: migliora le prestazioni del programma

### 8.3 Esempio — `exampleFunction` in Java [pp. 24–29]

Il CfG viene costruito suddividendo il codice in **blocchi base** (sequenze di istruzioni senza ramificazioni interne):

- Ogni blocco è un nodo del grafo
- Le condizioni (`if`, `while`, ecc.) generano **archi multipli**
- Modificando il codice (es. introducendo un errore), il CfG cambia e può rivelare **anomalie strutturali**

> [!example] Errore rilevabile tramite CfG
> Quando un tool di analisi statica (SpotBugs, SonarQube, PMD) analizza il codice, può segnalare:
> _"Possibile dereferenziazione di un valore null alla riga …"_
> Categoria: `NP_NULL_DEREFERENCE`

## 9. Java Inspection Checklist [checklist, 3 pp.]

> [!abstract] Fonte
> _Java Inspection Checklist_ — Copyright © 1999 Christopher Fox.

La checklist per l'ispezione di codice Java comprende **14 categorie** di difetti:

### 9.1 Difetti di dichiarazione (VC) [checklist p. 1]

- Nomi descrittivi conformi alle **naming conventions**?
- Variabili con nomi **confusamente simili**?
- Ogni variabile **correttamente tipata** e **inizializzata**?
- Variabili non-local che potrebbero essere **locali**?
- Variabili loop `for` dichiarate **nell'header**?
- Costanti letterali che dovrebbero essere **costanti nominate**?
- Attributi con **access modifier** appropriato (`private`, `protected`, `public`)?

### 9.2 Difetti di definizione metodi (FD) [checklist p. 1]

- Nomi descrittivi conformi alle convenzioni?
- Ogni parametro **controllato** prima dell'uso?
- Valore di ritorno **corretto** ad ogni punto di return?
- **Access modifier** appropriati?

### 9.3 Difetti di definizione classi (CD) [checklist p. 1]

- **Costruttori e distruttori** appropriati?
- Membri comuni nelle sottoclassi che dovrebbero stare nella **superclasse**?
- Gerarchia di ereditarietà **semplificabile**?

### 9.4 Difetti di riferimento dati (DR) [checklist p. 1]

- Ogni indice di array entro i **limiti** definiti?
- Ogni riferimento a oggetto/array **non-null**?

### 9.5 Difetti di calcolo (CN) [checklist p. 1]

- Computazioni con **tipi misti**?
- **Overflow/underflow** possibile?
- Assunzioni su **ordine di valutazione e precedenza** corrette?
- **Parentesi** usate per evitare ambiguità?

### 9.6 Difetti di confronto (CR) [checklist p. 1]

- Condizione booleana **corretta**?
- Operatori di confronto **corretti**?
- Espressioni booleane **semplificate**?
- Scambio involontario `&`/`&&` o `|`/`||`?

### 9.7 Difetti nel flusso di controllo (CF) [checklist p. 2]

- **Costrutto di loop** migliore?
- Tutti i loop **terminano**?
- Exit multipli dal loop **necessari e gestiti**?
- `switch` con **`default` case**?
- `break` mancanti nel `switch` **corretti e commentati**?
- Nesting di loop e branch **non troppo profondo**?
- **Eccezioni** gestite appropriatamente?

### 9.8 Difetti di I/O (IO) [checklist p. 2]

- File **aperti** prima dell'uso e **chiusi** dopo?
- Attributi dell'input coerenti con l'uso del file?
- Eccezioni I/O gestite **ragionevolmente**?

### 9.9 Difetti di interfaccia modulo (MI) [checklist p. 2]

- Numero, ordine, tipi e valori dei parametri in **accordo** con la dichiarazione?
- Unità di misura **concordanti**?
- Oggetti/array passati: modificati **correttamente**?

### 9.10 Difetti nei commenti (CM) [checklist p. 2]

- Header comment per ogni metodo, classe e file?
- Commenti per ogni dichiarazione?
- Commenti e codice **concordano**?
- Commenti **sufficienti** (né troppi né troppo pochi)?

### 9.11 Difetti di layout (LP) [checklist p. 2]

- **Indentazione standard** usata consistentemente?
- Ogni metodo ≤ ~**60 righe**?
- Ogni modulo ≤ ~**600 righe**?

### 9.12 Difetti di modularità (MO) [checklist p. 2]

- **Basso accoppiamento** tra moduli?
- **Alta coesione** dentro ogni modulo?
- Codice ripetitivo che potrebbe essere **estratto** in un metodo?
- **Librerie Java** usate quando appropriato?

### 9.13 Difetti di uso della memoria (SU) [checklist p. 3]

- Array **sufficientemente grandi**?
- Riferimenti a oggetti/array settati a **`null`** quando non più necessari?

### 9.14 Difetti di performance (PE) [checklist p. 3]

- Strutture dati o algoritmi **più efficienti** disponibili?
- Test logici ordinati: **economici e probabili prima**?
- Computazioni **spostabili fuori** dal loop?
- Loop brevi **srotolabili** (_unroll_)?
- Due loop sugli stessi dati **combinabili** in uno?

> **Fine del Blocco 2** — [pp. 18–29 + checklist 3 pp.] ✅

## 10. Data Flow Analysis (DfA) [pp. 30–35]

### 10.1 Definizione [p. 30]

La **Data Flow Analysis** analizza l'evoluzione del valore delle variabili durante l'esecuzione di un programma, permettendo di rilevare **anomalie**.

- Intrinsecamente è **dinamica**, ma alcuni aspetti possono essere analizzati **staticamente**.

### 10.2 Operazioni su una variabile [p. 31]

L'analisi statica DfA è legata a tre operazioni fondamentali:

| Operazione       | Simbolo | Significato                                                     |
| ---------------- | ------- | --------------------------------------------------------------- |
| **Definizione**  | `d`     | Alla variabile è assegnato un valore                            |
| **Uso**          | `u`     | Il valore della variabile è usato in un'espressione o predicato |
| **Annullamento** | `a`     | Al termine di un'istruzione il valore non è più significativo   |

> [!example] Esempio
> Nell'espressione `x := y + z`:
>
> - La variabile `x` è **definita** (`d`)
> - Le variabili `y` e `z` sono **usate** (`u`)

**Regole di sovrascrittura:**

- Una **definizione** (`d`) cancella l'effetto di una precedente definizione (nuovo valore)
- Un **annullamento** (`a`) cancella l'effetto di una precedente definizione (valore nullo)

### 10.3 Sequenze valide e anomale [p. 32]

Una corretta sequenza di operazioni prevede:

| Regola                                                                                     | Sequenza | Stato                                       |
| ------------------------------------------------------------------------------------------ | -------- | ------------------------------------------- |
| L'**uso** sia sempre preceduto da una **definizione** senza annullamenti intermedi         | `du`     | ✅ Valida                                   |
| Una **definizione** sia seguita da un **uso** prima di un'altra definizione o annullamento | `dud`    | ✅ Valida                                   |
| **Doppia definizione** senza uso intermedio                                                | `dd`     | ❌ Anomala — la prima definizione è inutile |
| **Definizione** seguita da **annullamento** senza uso                                      | `da`     | ❌ Anomala — valore mai utilizzato          |
| **Annullamento** seguito da **uso**                                                        | `au`     | ❌ Anomala — variabile con valore "sporco"  |

> [!warning] Sequenze anomale
> Le sequenze `dd`, `da` e `au` sono indicative di **difetti** nel codice: variabili definite inutilmente, valori mai letti, o uso di variabili non inizializzate.

### 10.4 DfA applicata al CfG [pp. 33–35]

Ad ogni nodo del CfG è possibile associare l'insieme delle variabili in esso **definite, usate e annullate**.

Per ogni variabile si determina la sequenza `[d, u, a]` facendo riferimento a tali insiemi e si valutano le **sequenze anomale**.

> [!example] Anomalie tipiche su CfG
>
> - Anomalia `dd` per la variabile `a` → doppia definizione senza uso intermedio
> - Anomalia `a→u` per la variabile `b` → uso dopo annullamento (possibile errore di assegnamento)

## 11. Controlli e vulnerabilità rilevabili [p. 36]

### 11.1 Controlli tramite CfG + DfA [p. 36]

L'analisi statica combinata (CfG + DfA) permette di individuare:

- Istruzioni **non raggiungibili**
- Variabili **usate prima** di essere inizializzate
- Variabili **dichiarate e mai usate**
- Variabili **scritte e mai lette**
- Variabile **mai usata** tra due assegnamenti
- Possibili **violazioni di array**

### 11.2 Vulnerabilità di sicurezza rilevabili [p. 36]

> [!warning] Vulnerabilità trovabili con analisi statica
>
> 1. **SQL Injection**
> 2. **Cross-Site Scripting (XSS)**
> 3. **Buffer Overflow**
> 4. **Remote Code Execution (RCE)**

## 12. SAST — Static Application Security Testing [pp. 37–38]

### 12.1 Definizione [p. 37]

> [!abstract] Definizione
> **SAST** = Analisi statica del codice sorgente per trovare le vulnerabilità che espongono ad attacchi.

Gli strumenti SAST forniscono agli sviluppatori:

- **Feedback in tempo reale** durante la codifica, suggerendo la risoluzione dei problemi prima della compilazione
- **Rappresentazioni grafiche** dell'analisi, segnalando la posizione delle vulnerabilità
- Una **guida approfondita** su come risolvere i problemi

### 12.2 Limiti [p. 37]

> [!warning] Limiti degli strumenti SAST
>
> - Devono **conoscere le peculiarità** e le vulnerabilità dei linguaggi di programmazione usati
> - Devono avere meccanismi di **aggiornamento continuo** delle regole di rilevazione dei modelli non-compliant

### 12.3 Tool SAST [p. 38]

| Tool                              | Best for                                                                                                 |
| --------------------------------- | -------------------------------------------------------------------------------------------------------- |
| **Checkmarx SAST** e **Veracode** | Aziende medie/grandi — robusto supporto multilingue (Java, JavaScript, Python) e integrazione **CI/CD**  |

## 13. Prova di correttezza del software [pp. 39–44]

### 13.1 Triple di Hoare [p. 39]

La prova di correttezza è la tecnica di analisi statica **più completa**.

**Obiettivo:** dimostrare che l'implementazione soddisfa la sua specifica.

La specifica viene espressa formalmente in termini di due **asserzioni**:

$$\{P\}\ S\ \{Q\}$$

| Simbolo | Significato                                                  |
| ------- | ------------------------------------------------------------ |
| $P$     | **Precondizione** — vale prima dell'esecuzione del programma |
| $S$     | **Programma** (o frammento di codice)                        |
| $Q$     | **Postcondizione** — vale dopo l'esecuzione del programma    |

> [!info] Tripla di Hoare
> Si dimostra che il programma $S$ **trasforma** la precondizione $P$ nella postcondizione $Q$.

### 13.2 Esempio semplice [p. 40]

```
S: n := n + 9
P: n > 5
Q: n > 14
```

Correttezza: $\{n > 5\}\ \ n := n + 9\ \ \{n > 14\}$ ✅

### 13.3 Esempio: contratto di SQRT [p. 41]

Sia `SQRT` una routine per il calcolo della radice quadrata:

- **$S$:** algoritmo per calcolare $y$ come radice quadrata di $x$ (con precisione $\varepsilon$)
- **$P$:** $x \geq 0$
- **$Q$:** $|y^2 - x| \leq \varepsilon$

$$\{x \geq 0\}\ \ \text{SQRT}\ \ \{|y^2 - x| \leq \varepsilon\}$$

### 13.4 Limiti della prova di correttezza [p. 42]

> [!warning] Limiti
>
> - Le prove sono molto **formali** e difficili da costruire per il programmatore medio.
> - La dimostrazione è **complessa** e difficilmente effettuabile con strumenti automatici.
> - Eccezione: quando il **contratto è immerso nel codice** (caso dei contratti **JML** o in generale del **DbC**).

## 15. Correttezza di una classe nel DbC [p. 44]

Nel **Design by Contract** una classe è corretta se sono corretti:

### 15.1 Operazione di creazione

Per ogni nuova istanza, l'**invariante** (INV) deve valere dopo il costruttore:

$$\{PRE_{costr}\}\ \ \text{constructor}\ \ \{INV \wedge POST_{costr}\}$$

### 15.2 Ogni altro metodo OP

Se si chiama un metodo `OP` con precondizione vera (e INV), dopo l'esecuzione vale la postcondizione e continua a valere l'invariante:

$$\{PRE_{op} \wedge INV\}\ \ \text{OP}\ \ \{POST_{op} \wedge INV\}$$

> [!tip] Invariante di classe
> L'**invariante** è il "filo conduttore" che lega tutti i metodi: deve valere **prima e dopo** ogni chiamata pubblica.

## 16. Abstract Interpretation [pp. 45–47]

### 16.1 Concetto [p. 45]

Tecnica di analisi statica basata sul concetto di **astrazione**:

1. Partendo dalle istruzioni del codice, viene determinato il **dominio astratto** del valore delle variabili
2. Si definisce lo **stato astratto** ad ogni istruzione (fino allo stato finale)
3. L'analisi dello stato astratto permette di rilevare:

| Proprietà verificabile             |
| ---------------------------------- |
| Assenza di `NullPointerException`  |
| Limiti di array **rispettati**     |
| Assenza di **divisioni per zero**  |
| Variabili sempre **inizializzate** |
| Assenza di **dead code**           |
| **Sicurezza dei puntatori**        |

### 16.2 Simboli per domini astratti [p. 46]

| Simbolo        | Significato                              |
| -------------- | ---------------------------------------- |
| `NEG`          | Valore negativo                          |
| `ZERO`         | Valore zero                              |
| `POS`          | Valore positivo                          |
| `TOP` ($\top$) | Sconosciuto — può essere qualsiasi segno |
| $\bot$         | Non definito / non inizializzato         |

### 16.3 Esempio [p. 46]

> [!example] Analisi astratta di una divisione
> **Stato iniziale:** $x: \top$, $y: \bot$
>
> **Ramo `then`:** $x: POS$, $y: POS$
>
> **Ramo `else`:** $x: ZERO \cup NEG$, $y: NEG$
>
> **Join degli stati:** $x: \top$, $y: POS \cup NEG$
>
> **Stato finale:** $x: \top$, $y: POS \cup NEG$, $z: POS \cup NEG$
>
> **Conclusione statica:** la divisione `10 / y` è **sicura** — non ci sarà divisione per zero (perché $y$ è $POS$ o $NEG$, mai $ZERO$).
