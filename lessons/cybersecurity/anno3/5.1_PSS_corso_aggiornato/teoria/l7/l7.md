## 2. Macchine a stati finiti in Java

### 2.1 Ripasso delle FSM

Le macchine a stati finiti (FSM) sono state già introdotte nell'unità didattica dedicata e vengono usate spesso per modellare e simulare hardware e sistemi di controllo. Una FSM è definita da: [p. 4]

- un **insieme di stati**;
- **transizioni tra stati**, dove ciascuna transizione è determinata dall'input (evento) e dallo stato corrente;
- ogni transizione può produrre un **output** (azione).

La domanda che ci si pone ora è: come si possono tradurre le FSM in codice Java?

### 2.2 Esempio: il sistema Turnstile (tornello)

L'esempio di riferimento è un tornello con due stati — **Locked** e **Unlocked** — e due eventi — **coin** e **pass**. La tabella delle transizioni è la seguente: [p. 5]

| Stato corrente | Evento | Stato successivo | Azione |
|---|---|---|---|
| Locked | coin | Unlocked | unlock |
| Locked | pass | Locked | alarm |
| Unlocked | coin | Unlocked | thankyou |
| Unlocked | pass | Locked | lock |

## 3. I Design Pattern

### 3.1 Pattern del software

Un **pattern del software** è una soluzione provata e applicabile a un particolare problema di progettazione. Il pattern descrive la soluzione in una forma standard, in modo che possa essere facilmente condivisa e riusata. Una descrizione strutturata di un pattern comprende: nome, contesto (la situazione in cui può essere applicato), problema, soluzione e conseguenze (risultati positivi e negativi, compromessi). [p. 6]

### 3.2 Design pattern

Un **design pattern** (nel senso della *Gang of Four* — GoF) è la descrizione di oggetti e classi che comunicano, personalizzati per risolvere un problema generale di progettazione in un contesto particolare. I design pattern vengono impiegati nell'ambito della **progettazione di dettaglio** dei componenti. Esempi classici includono Singleton, State, Adapter, Proxy, Observer, e altri. [p. 7]

## 4. Lo State Pattern

### 4.1 Definizione

Lo **State Pattern** è un design pattern con il seguente scopo: permettere a un oggetto di cambiare il suo comportamento quando il suo stato interno cambia. [p. 8]

> [!info] State Pattern
> **Contesto di applicazione:** quando il comportamento di un oggetto dipende da un suo stato e tale comportamento si esprime tipicamente come un'istruzione condizionale sullo stato e sull'evento ricevuto.
>
> **Soluzione proposta:** definire una classe concreta per ogni stato. Le classi di stato implementano un'interfaccia comune che reagisce a ogni possibile input. Gli oggetti di stato possono richiamare azioni sull'oggetto che li controlla e decidono autonomamente il prossimo stato (cioè le transizioni di stato).

### 4.2 Schema delle classi

Lo schema generale dello State Pattern prevede tre elementi principali: [pp. 9–10]

1. Una **classe Context** che rappresenta la macchina a stati finiti, con un riferimento allo stato corrente (`state`).
2. Un'**interfaccia State** che dichiara i metodi per reagire a ogni possibile evento.
3. **Classi concrete** (StateA, StateB, …) che implementano l'interfaccia e definiscono il comportamento specifico di ogni stato.

Il Context delega la risposta a ogni evento allo stato corrente tramite polimorfismo. Quando lo stato cambia, il Context punta a un'istanza diversa (ma sempre di tipo State). [p. 10]

> [!abstract] Vantaggi dello State Pattern
> - Il codice è più **modulare** (anche se più verboso).
> - Il codice è più **pulito**, senza grandi blocchi condizionali.
> - L'**estensione** (aggiunta di nuovi stati) non comporta la modifica del codice esistente.

### 4.3 Turnstile con lo State Pattern

Applicando lo State Pattern al tornello, lo schema delle classi diventa: [pp. 11–12]

- **`TurnstileFSM`**: rappresenta la FSM del tornello. Contiene un riferimento allo stato corrente (`state` di tipo `TurnstileState`) e un riferimento al controller (`TurnstileController`). Delega la gestione degli eventi allo stato corrente e la gestione delle azioni al controller.
- **`TurnstileState`** (interfaccia): dichiara i metodi `coin(TurnstileFSM)` e `pass(TurnstileFSM)`.
- **`LockedState`** e **`UnlockedState`**: implementano `TurnstileState`, ciascuna definendo il comportamento per entrambi gli eventi.
- **`TurnstileController`**: gestisce le azioni fisiche del tornello (`lock()`, `unlock()`, `thankyou()`, `alarm()`).

### 4.4 Implementazione della classe `TurnstileFSM`

La classe che rappresenta la FSM del tornello delega allo stato corrente la gestione degli eventi e al controller la gestione delle azioni: [p. 13]

```java
class TurnstileFSM {
    private TurnstileController turnstileController;
    private TurnstileState state = ...;

    // delega allo stato corrente l'evento
    public void coin() { state.coin(this); }
    public void pass() { state.pass(this); }

    // agisci sul controller quando richiesta azione
    void thankyou() { turnstileController.thankyou(); }
    void alarm()    { turnstileController.alarm(); }
    void lock()     { ... }
    void unlock()   { ... }
}
```

### 4.5 La gerarchia di stati

Ogni sottoclasse implementa l'interfaccia `TurnstileState` e definisce un **oggetto costante** (singoletto — *Singleton Pattern*) che rappresenta quello stato: [pp. 14–15]

```java
interface TurnstileState {
    void coin(TurnstileFSM t);
    void pass(TurnstileFSM t);
}

class LockedState implements TurnstileState {
    public static TurnstileState LOCKEDSTATE = new LockedState();

    public void coin(TurnstileFSM t) { ... }
    public void pass(TurnstileFSM t) { ... }
}

class UnlockedState implements TurnstileState { ... }
```

### 4.6 Singolo metodo in uno stato

Il cambiamento di stato e l'azione al verificarsi dell'evento vengono gestiti nel metodo corrispondente dello stato corrente. Ad esempio, l'evento `coin` nello stato `LockedState` causa la transizione a `UnlockedState` e l'azione `unlock`: [p. 16]

```java
class LockedState ... {
    public void coin(TurnstileFSM t) {
        t.setState(UnlockedState.UNLOCKEDSTATE);
        t.unlock();
    }
}
```

> [!tip] Nota implementativa
> Per azionare la FSM (cambiare stato e invocare azioni), è necessario passare l'oggetto FSM come parametro al metodo dello stato.

### 4.7 Estensibilità dello State Pattern

L'aggiunta di nuovi elementi è semplice e ben localizzata: [p. 17]

- **Aggiungere uno stato NS**: si crea una nuova sottoclasse di `TurnstileState` che implementa i metodi per tutti gli eventi e definisce una costante per il singoletto.
- **Aggiungere un evento EV**: si aggiunge un metodo `EV` all'interfaccia `TurnstileState`; tutti gli stati esistenti dovranno definire il comportamento per il nuovo evento.

## 5. FSM dell'orologio digitale e sintesi dello State Pattern

### 5.1 FSM dell'orologio digitale

Le slide presentano un secondo esempio di FSM: un **orologio digitale** con tre stati (Off, Normal, Setting) e quattro eventi (set, io, m, h). L'orologio dispone di azioni come `display()`, `displaylamp()`, `setmin()` e `sethour()`, delegate a un controller `FSMcontroller`. Lo State Pattern viene applicato con la stessa struttura vista per il tornello: la classe `FSMclock` rappresenta la FSM, l'interfaccia `FSMstate` dichiara i metodi per ogni evento, e le classi `Off`, `Normal` e `Setting` implementano i comportamenti specifici. [pp. 18–19]

La codifica concreta (disponibile su Ariel) segue fedelmente il pattern: ogni stato è una classe con un singoletto statico, e ogni metodo decide la transizione e l'azione da eseguire.

### 5.2 Sintesi: implementare una FSM con lo State Pattern

In sintesi, per implementare una FSM in Java usando lo State Pattern: [p. 20]

> [!abstract] Riepilogo dello State Pattern per FSM
> - La FSM è rappresentata da una **classe** che delega le azioni a un **controller** e le transizioni di stato a una **gerarchia di stati**.
> - Ogni possibile stato è una **sottoclasse** di un'interfaccia e implementa un **metodo per ogni evento** (input possibile).
> - Nel metodo del singolo stato c'è il codice che gestisce il **cambio di stato** e l'**azione** (output) quando la FSM si trova in quello stato.

### 5.3 Tool automatici per FSM

Esistono numerosi tool che permettono di disegnare (o scrivere) una FSM e poi generano automaticamente tutto il codice: [p. 21]

- **SMC** (*The State Machine Compiler*): prende una descrizione testuale della macchina e produce codice C, C++, Java, PHP, ecc.
- **YAKINDU Statechart Tools**: il più performante, permette di definire statechart graficamente e genera codice Java eseguibile.

Le slide mostrano esempi in YAKINDU, tra cui un **simple light switch** e un **controller per due luci** con variabili `light1` e `light2` di tipo `Light`, inizializzate a valori diversi di massima luminosità. Quando la state machine `LightController` inizia, anche le state machine delle due luci si avviano. Il codice Java generato da YAKINDU può essere eseguito direttamente. [pp. 22–26]

## 6. Vulnerabilità del software: la sicurezza dei tipi

### 6.1 L'importanza della sicurezza dei tipi

La sicurezza dei tipi in un programma è un aspetto cruciale. Se l'esecutore (il PC o la macchina virtuale) non riesce a distinguere i tipi di un programma, può facilmente causare errori. Molti attacchi informatici sfruttano proprio debolezze nel controllo dei tipi di linguaggi diffusi come il C. Come affermava **Robert Milner**: [p. 28]

> [!quote] "Well-typed programs never go wrong."
> — Robert Milner

### 6.2 Che cos'è un tipo

Un **tipo** è un insieme di valori omogenei insieme alle operazioni che si possono effettuare su di essi (con risultato di tipo omogeneo). [p. 29]

> [!example] Esempi e non-esempi di tipi
> **Tipi validi:** tipi semplici (Integer, String), tipi strutturati (classi), funzioni (`int → bool` — una funzione che dato un intero restituisce un booleano; anche le funzioni e i metodi definiscono un tipo).
>
> **Non-tipi:** numeri dispari, array contenenti sia String che Integer (dipende però dal linguaggio di programmazione).

### 6.3 A cosa servono i tipi

I tipi servono a tre scopi fondamentali: [p. 30]

1. **Organizzare e rappresentare** gli elementi del dominio del problema (documentazione). Spesso i tipi corrispondono ai concetti nel dominio e il loro uso precisa cosa si intende fare con certi identificatori, consentendo al compilatore di effettuare controlli.
2. **Istruire il compilatore** su come rappresentare i dati in memoria (ad esempio, `short` richiede meno bit di `int`).
3. **Assicurarsi** che le sequenze di bit in memoria siano interpretate correttamente, evitando errori come `3 + true + "Angelo"`.

### 6.4 Errori di tipo: conseguenze a livello hardware

Confondere dati con funzioni o confondere tipi di dati semplici può avere conseguenze gravi a livello hardware: [p. 31]

> [!warning] Errori hardware da violazione dei tipi
> - **Confondere dati con funzioni**: se `x` è definita come intero e si chiama `x()`, si tenta di eseguire un dato caricando nella CPU codici potenzialmente non corretti, causando un errore hardware.
> - **Confondere tipi di dati semplici**: eseguire `float_add(3, 4.5)` dove `3` è di tipo `int` fa sì che la CPU chiami una routine della FPU interpretando la sequenza di bit di `3` come un float, generando potenzialmente un errore hardware.

### 6.5 Errori di tipo: conseguenze semantiche

Oltre agli errori hardware, le violazioni dei tipi possono causare errori **semantici**, in cui il programma fa qualcosa di diverso da ciò che dovrebbe fare: [p. 32]

> [!example] Esempi di errori semantici
> **Con tipi primitivi:** `int_add(3, 4.5)` — la sequenza di bit che rappresenta `4.5` viene interpretata come `int`, con perdita di valore.
>
> **Con oggetti ed ereditarietà in Java:** data `class Quadrato extends Figura`:
> - `Figura a1 = new Quadrato()` — **OK**: un quadrato è una figura.
> - `Quadrato b1 = new Figura()` — **NO**: `Quadrato` potrebbe avere metodi aggiuntivi che si potrebbero invocare su `b1`, ma che non esistono perché `b1` è in realtà una `Figura`.

### 6.6 Type safety

Un linguaggio di programmazione $L$ si dice **type safe** se non esiste alcun programma scritto in $L$ che possa violare la distinzione dei tipi in $L$. Esempi di violazioni dei tipi includono: confondere interi e float, chiamare una funzione attraverso un intero, accedere a una zona di memoria sbagliata (*memory unsafe*). [p. 33]

> [!info] Sicurezza dei linguaggi di programmazione
>
> | Safety | Linguaggio | Motivo |
> |---|---|---|
> | **Non safe** | C e C++ | Type cast non controllato, aritmetica dei puntatori |
> | **Quasi safe** | Pascal | Deallocazione esplicita e dangling pointers |
> | **Safe** | Java, Python | Controllo completo dei tipi |

> **Fine del Blocco 2** — [pp. 18–33] ✅

## 7. Type checking: quando si controllano i tipi?

Tra i linguaggi type safe si distinguono due categorie a seconda del **momento** in cui avviene il controllo dei tipi: [p. 35]

> [!info] Due approcci al type checking
> - **Run-time type checking**: il controllo avviene durante l'esecuzione. Esempio: Python è un linguaggio interpretato che controlla i tipi a run-time.
> - **Compile-time type checking**: il controllo avviene durante la compilazione. Esempi: ML, Haskell, ADA sono linguaggi con meccanismi di *secure static typing*.

La classificazione risultante è: [p. 36]

| | Type safe | Non type safe |
|---|---|---|
| **Run-time type check** | Python | — |
| **Compile-time type check** | Java | C, C++ |

### 7.1 Java: un approccio ibrido

Java utilizza il type checking in **tre momenti distinti**: static type checking (a tempo di compilazione), verification time (verifica del bytecode) e run-time. Il compilatore Java effettua il controllo statico, ma laddove non è possibile determinare la sicurezza dei tipi in compilazione, introduce un **controllo a run-time**. [p. 37]

> [!example] Esempio: conversione di tipo in Java
> ```java
> class Quadrato extends Figura
> Figura b;
> Quadrato a = (Quadrato) b;
> ```
> La conversione al sottotipo `Quadrato` è corretta solo se `b` è effettivamente un'istanza di `Quadrato` (o di una sua sottoclasse). Tale controllo non si può fare in compilazione: il compilatore introduce quindi un controllo da eseguire durante l'esecuzione per verificare che `b` sia convertibile a `Quadrato`. [p. 38]

### 7.2 Pro e contro dei due approcci

Entrambi gli approcci prevengono errori di tipo, ma con compromessi diversi: [pp. 39–40]

- **Run-time checking** rallenta l'esecuzione, perché le conversioni di tipo vengono controllate ogni volta.
- **Compile-time checking** limita la flessibilità dei programmi: tutte le istruzioni, anche quelle mai eseguite, devono essere corrette.

> [!example] Programma run-time safe ma non compile-time safe
> ```java
> int x;
> if (0 > -1) { x++; } else { x = "ciao"; }  // NON compila
> ```
> In Java questo codice non compila perché il ramo `else` assegna a `x` (di tipo `int`) una `String`. Eppure il programma è **runtime type safe**: nessuna esecuzione causerebbe effettivamente l'errore di tipo, poiché `0` è sempre `> -1`. [p. 40]

## 8. Il linguaggio C non è safe

### 8.1 Caratteristiche insicure del C

Alcune caratteristiche del linguaggio C e C++ possono dare origine a errori di sicurezza: [p. 41]

> [!warning] Vulnerabilità del C/C++
> 1. **Dereferenziazione del null**
> 2. **Type cast non controllato**
> 3. **Pointer arithmetic** (aritmetica dei puntatori)
> 4. **Accesso alla memoria non valida**:
>    - *Violazione spaziale*: out of bound, buffer overflow
>    - *Violazione temporale*: dangling pointers
>
> Queste caratteristiche rendono il C molto flessibile e veloce, ma **a discapito della sua sicurezza**. È responsabilità del programmatore evitare l'introduzione di difetti.

### 8.2 Dereferenziazione di null

La **dereferenziazione** è l'operazione di accesso alla memoria tramite un puntatore. In C, tale operazione non viene controllata: se si accede a una cella puntata da un puntatore nullo, si ottiene un **segmentation fault** (errore del sistema operativo). [p. 42]

```c
int main() {
    int *ptr;
    ptr = 0;
    *ptr = 2;  // segmentation fault
}
```

### 8.3 Type cast non safe

Il C permette la **conversione non controllata** da un tipo a un altro. Questo può causare perdita di informazioni (conversione da tipo a sopratipo) o comportamenti pericolosi (conversione da intero a funzione, che tenta di eseguire una locazione di memoria arbitraria). [p. 43]

```c
double d;
int i;
...
i = d;  // possibile perdita di informazioni
```

### 8.4 Aritmetica dei puntatori

In C i puntatori sono trattati come interi. Mediante l'aritmetica dei puntatori è possibile accedere a zone di memoria con tipi diversi da quello dichiarato: se il puntatore `p` è di tipo `A*`, l'espressione `*(p+i)` ha formalmente tipo `A`, ma il valore effettivamente memorizzato a `p+i` potrebbe avere **qualsiasi tipo**. L'assegnamento `x = *(p+i)` con `x` di tipo `A` permette di memorizzare in `x` un valore senza alcun controllo di tipo. [p. 44]

### 8.5 Violazioni di memoria (spatial)

Mediante i puntatori si può facilmente accedere alla memoria in modo scorretto: [pp. 45–46]

```c
void f(int *p, int i, int v) {
    p[i] = v;
}
```

In questo modo si accede all'indirizzo `p+i`, che potrebbe contenere dati importanti o altro codice. Le conseguenze possibili sono gravi: modificare il **return address** di una chiamata di procedura per eseguire altro codice, modificare dei diritti, o leggere informazioni riservate. Questo è il meccanismo tipico del **buffer overflow**.

### 8.6 Violazioni di memoria (conversione int → puntatore)

I puntatori in C sono assimilati a interi. Tramite il cast di dati interi a puntatori, si può accedere a una zona di memoria a piacere: [p. 47]

```c
int main() {
    char *p;
    p = 1000;    // p punta all'indirizzo 1000
    *p = 'a';    // scrive 'a' all'indirizzo 1000
}
```

Questo programma è sintatticamente corretto in C, ma è estremamente pericoloso.

### 8.7 Deallocazione esplicita e dangling pointers

In C una locazione puntata da un puntatore `p` può essere **deallocata** (liberata) esplicitamente dal programmatore. Dopo la deallocazione, `p` diventa un **dangling pointer**: punta a una zona di memoria che è stata liberata e potrebbe essere riutilizzata dal sistema operativo per memorizzare un altro tipo di valore. Continuando a usare `p`, si può accedere a questa memoria e violare la type safety. [p. 48]

### 8.8 Dangling pointers sullo stack

Un caso frequente di dangling pointer si verifica quando si usano puntatori a celle dello **stack**. Questo accade quando si crea un puntatore `p` a una zona di memoria `A` locale a un metodo (ad esempio variabili locali), allocata sullo stack. Quando il metodo termina, `A` viene liberata e `p` diventa un dangling pointer. [p. 49]

> [!example] Esempio: funzione `itoa` con dangling pointer
> ```c
> char* itoa(int i) {
>     char buf[20];
>     sprintf(buf, "%d", i);
>     return buf;  // buf punta a un array locale deallocato!
> }
> ```
> La funzione converte un intero nella stringa corrispondente restituendo il puntatore alla stringa ottenuta. Tuttavia, `buf` punta a un array locale che viene deallocato all'uscita della funzione: `buf` è un **dangling pointer**. [p. 50]

> **Fine del Blocco 3** — [pp. 34–50] ✅

## 9. Soluzioni ai dangling pointers

Come si possono evitare i dangling pointers? Le slide propongono due soluzioni: [p. 51]

1. **Evitare di puntare zone di memoria sullo stack** e usare la `malloc`, che crea puntatori a zone sicure (heap):
   ```c
   struct Point *result = malloc(sizeof(struct Point));
   ```
2. **Uso del garbage collector (GC)** invece della deallocazione esplicita: il GC marca automaticamente le zone da liberare e che possono essere riutilizzate.

## 10. Cosa fare per scrivere codice safe?

Se si vuole scrivere codice **safe**, le opzioni sono: scrivere attentamente, progettare prima di codificare, documentare, ecc. Se si vuole essere **certi** che il codice sia safe, esistono due soluzioni: [p. 52]

1. Usare **linguaggi type safe** (Java, Python, …) e linguaggi più astratti.
2. Usare linguaggi come il C insieme a **tool** che aiutano a rendere i programmi C safe.

### 10.1 Svantaggi dei linguaggi più astratti

L'uso di linguaggi safe come Java comporta un prezzo: [p. 53]

> [!warning] Costi dei linguaggi type safe
> 1. **Prestazioni inferiori**: per il controllo dei limiti nell'accesso agli array, il garbage collection per evitare dangling pointers.
> 2. **Impiego di maggiore memoria**: per mantenere informazioni sui tipi, sulla dimensione degli array.
> 3. **Annotazione dei tipi**: maggiore verbosità nelle dichiarazioni.
> 4. **Porting di codice** già esistente in C (per quanto Java abbia sintassi simile).

### 10.2 Vantaggi del C

Il C è tutt'oggi usato per molte applicazioni come il **sistema operativo** e i **device drivers** grazie a: prestazioni elevate, gestione esplicita della memoria, controllo della rappresentazione dei dati a basso livello, e possibilità di riuso del codice esistente già scritto in C. [p. 54]

### 10.3 Tipiche vulnerabilità del C

Le vulnerabilità del C che possono causare problemi di sicurezza si classificano in: [p. 55]

| Categoria | Esempi |
|---|---|
| Errori **spaziali** di accesso alla memoria | Out of bound access, buffer overflow |
| Errori **temporali** di accesso alla memoria | Dangling pointers |
| Errori di **cast** | Tra diversi tipi di puntatori, tra puntatori e dati interi, tipi unione |
| **Memory leaks** | Programmi che non rilasciano la memoria quando non serve più |

### 10.4 Come rendere il C safe?

Esistono tre approcci principali: [p. 56]

1. **Tool per l'analisi statica e dinamica** per trovare safety violations (es. Purify della Rational/IBM).
2. **Librerie** per rendere i programmi C safe.
3. **Tool e linguaggi** con due approcci distinti:
   - (a) Rendere sicuri programmi C esistenti: **SafeC**, **CCured**.
   - (b) Varianti safe del C: **Cyclone**, **Vault**.

### 10.5 Analisi dinamica con Purify

**Purify** è uno strumento di analisi dinamica sviluppato da Rational/IBM: [p. 57]

| Aspetto | Dettaglio |
|---|---|
| **Approccio** | Analisi dinamica |
| **Input** | Programmi C/C++ qualsiasi |
| **Output** | Eseguibili linkati con Purify |
| **Metodo** | Inserimento di controlli per trovare durante l'esecuzione errori di accesso alla memoria o memoria non rilasciata |
| **Pro** | Si applica a codice già esistente |
| **Contro** | Rallenta l'esecuzione e non garantisce la scoperta di ogni errore |

### 10.6 Cyclone

**Cyclone** è un linguaggio C-like progettato per essere safe: [p. 58]

| Aspetto | Dettaglio |
|---|---|
| **Approccio** | Linguaggio safe simile al C |
| **Input** | Programmi C modificati |
| **Output** | Programmi C sicuri |
| **Metodo** | Controlli dinamici solo dove necessario e garbage collector |
| **Pro** | Minimo overhead di tempo e di memoria |
| **Contro** | Richiede di modificare i programmi originali |

> **Fine del Blocco 4** — [pp. 51–58] ✅

- Alfredo De Sanctis, *Il Modello di sicurezza in Java* (disponibile su Ariel, in 2 documenti).

### 11.2 Java e la JVM

Il flusso di lavoro in Java prevede: il codice sorgente (`.java`) viene compilato dal compilatore (`javac`) producendo un file `.class` (bytecode); la **Java Virtual Machine (JVM)** esegue il bytecode (`java prova.class`) utilizzando le librerie disponibili. È possibile anche usare direttamente la JVM senza passare per il compilatore (ad esempio eseguendo bytecode ricevuto dalla rete). [p. 61]

### 11.3 Meccanismi di sicurezza di Java

L'architettura di sicurezza di Java si basa su numerosi meccanismi: [p. 62]

> [!info] Meccanismi di sicurezza di Java
> - **Sicurezza dei tipi**: Java è type safe.
> - **Verifica del bytecode**: solo codice Java legittimo viene accettato.
> - **Controllo runtime delle classi caricate**: la JVM carica e controlla il codice dinamicamente.
> - **Accesso mediato alle risorse**: i programmi accedono alle risorse attraverso la JVM, non direttamente.

### 11.4 Diverse politiche di sicurezza

L'evoluzione del modello di sicurezza di Java si articola in due fasi principali: il **modello Java 1.0/1.1** e il **modello da Java 2** in poi. [p. 63]

## 12. Il modello di sicurezza Java 1.0

### 12.1 Applet e applicazioni

Nel modello di sicurezza di Java 1.0 si distinguono due categorie di codice: [p. 64]

- **Applet**: codice scaricato da Internet ed eseguito localmente. Si assume che sia **potenzialmente malizioso** e viene eseguito in un **sandbox** (ambiente isolato e controllato).
- **Applicazioni**: programmi sul filesystem locale, assimilabili ai normali programmi e considerati fidati.

### 12.2 Sandbox Policy per le applet

La politica del sandbox per un'applet in Java 1.0 impone le seguenti restrizioni: [p. 65]

> [!warning] Restrizioni del sandbox (Java 1.0)
> - Non può accedere direttamente al **file system locale** o ai devices.
> - Può connettersi **solo** con la sorgente da cui proviene l'applet.
> - Non può eseguire nessun **programma locale**.
> - Viene marcato come **untrusted** e viene visualizzata un'icona.
> - Non può manipolare (estendere, …) le **classi base** di Java.
>
> Le **applicazioni**, al contrario, hanno accesso **illimitato**.

## 13. Architettura della Sandbox

### 13.1 Il Bytecode Verifier

Il compilatore Java controlla già che il sorgente non violi le regole di sicurezza del linguaggio, ma non è detto che il bytecode sia stato prodotto dal compilatore — potrebbe essere stato generato o manipolato manualmente. Per questo motivo, il bytecode viene **verificato prima dell'esecuzione** per controllare che: [p. 67]

- non esegua conversioni illegali tra dati;
- non modifichi (*forge*) puntatori (riferimenti);
- non violi le restrizioni di accesso (es. accedere a un campo privato);
- usi in modo corretto gli oggetti per come sono dichiarati;
- chiami i metodi in modo corretto (argomenti corretti).

> [!tip] Efficienza della verifica
> La verifica del bytecode viene fatta **staticamente** (una sola volta, al caricamento). Gli stessi controlli potrebbero essere fatti a runtime dalla JVM, ma ciò rallenterebbe l'esecuzione: ad esempio, il controllo di un metodo andrebbe fatto ogni volta che è invocato. Grazie alla verifica statica, durante l'esecuzione la JVM può assumere che alcuni controlli siano già stati eseguiti. [p. 68]

### 13.2 SecurityManager e ClassLoader

In JDK 1.0, la sandbox è costituita, oltre che dal Bytecode Verifier, da due classi fondamentali: **SecurityManager** e **ClassLoader**. I tre componenti lavorano insieme per garantire la sicurezza. [p. 69]

### 13.3 SecurityManager

La classe `SecurityManager` definisce i **metodi di controllo dei permessi** di sicurezza chiamati dal sistema. Ad esempio, `checkRead(String filename)` verifica se la lettura di un file è permessa; in caso negativo, lancia un'eccezione. [p. 70]

In Java 1.0 è possibile creare una propria security policy creando una **sottoclasse** di `SecurityManager` e installandola con `System.setSecurityManager`. Una volta settata, non può essere revocata o cambiata.

### 13.4 ClassLoader

Il **ClassLoader** carica effettivamente le classi: dato il nome della classe, trova la sua definizione e istanzia una classe (o insieme di classi) in un **namespace**. [p. 71]

> [!info] Ruolo del ClassLoader
> - Ogni classe mantiene un riferimento all'istanza di ClassLoader che l'ha caricata.
> - Mantiene i **namespace** di applet diverse **separati** (mediante diverse istanze di ClassLoader).
> - Le classi nel `CLASSPATH` sono considerate **fidate** (in JDK 1.0–1.1); le classi di sistema sono sempre fidate.
> - Per le classi non fidate, invoca il **Bytecode Verifier** e il **SecurityManager** per i permessi.

### 13.5 Flusso di esecuzione della sandbox

Il flusso completo di sicurezza nella sandbox è il seguente: [pp. 72–73]

1. Il **ClassLoader** preleva il bytecode dal Web.
2. Lo **verifica** tramite il Bytecode Verifier.
3. Lo **istanzia** in una classe (o insieme di classi) in un namespace.
4. L'applet viene eseguito: quando un metodo viene invocato, il ClassLoader trova il riferimento alla classe.
5. Se il metodo è **potenzialmente pericoloso**, il **SecurityManager** viene consultato prima che il metodo venga eseguito.
6. Il SecurityManager esegue i **runtime-checks** sulla base dell'origine della classe chiamante, con l'autorità di **porre un veto** sull'operazione.

## 14. Da JDK 1.1 a Java 2: applet firmate e controllo fine degli accessi

### 14.1 Applet firmate in JDK 1.1

JDK 1.1 introduce il concetto di **applet firmate digitalmente**: un'applet dotata di firma digitale valida ottiene gli stessi permessi delle applicazioni locali, superando le restrizioni del sandbox. Il modello rimane però di tipo "tutto o niente": un'applet è o completamente fidata o completamente sandboxed. [p. 74]

### 14.2 Java 2: controllo fine degli accessi

**Java 2** supera il modello binario di JDK 1.0/1.1 introducendo un **controllo degli accessi a grana fine** (*fine-grained access control*) basato su **security policies** e **permessi**. Non si tratta più di un approccio "tutto o niente": alcune applet possono avere **alcuni diritti** per accedere ad **alcune risorse** fuori dal sandbox. [p. 75]

> [!example] Esempio di accesso fine
> Un'applet può avere il permesso di leggere un certo file in cui memorizza i dati, ma nessun altro file del sistema.

### 14.3 Security Policy in Java 2

La Security Policy di Java 2 associa a ogni **origine del codice** (CodeSource) un insieme di **permessi** (Permissions): [p. 76]

| Origine del codice (CodeSource) | | Permissions |
|---|---|---|
| **Base URL** | **Signature** | |
| `http://www.schwab.com/classes/stockeditor.jar` | Schwab's signature | Read/write file `/home/daw/stocks` |
| `http://*.schwab.com/` | (non richiesta) | Connect/accept `bankofamerica.com` ports 1–1023; Read file `/home/daw/logo.png` |

### 14.4 CodeSource e Permission

La classe `CodeSource` specifica **da dove proviene** il codice: l'URL di origine e un array di certificati. È immutabile. [p. 77]

La classe `Permission` rappresenta, con la sua gerarchia, i **permessi** disponibili:

- `FilePermission`: permessi di scrittura/lettura di file.
- `SocketPermission`: permessi per connettersi ai socket.
- E altre sottoclassi per tipologie specifiche di risorse.

### 14.5 Vantaggi dell'architettura di Java 2

L'architettura di sicurezza di Java 2 offre vantaggi significativi rispetto alle versioni precedenti: [p. 78]

> [!abstract] Vantaggi di Java 2
> - Non è necessario scrivere codice particolare per **modificare la Security Policy**.
> - **Separazione** tra l'espressione della Security Policy e la sua applicazione:
>   - L'espressione dei diritti avviene usando istanze della classe `Permission` e delle sue sottoclassi.
>   - L'applicazione della Security Policy è fatta dal `SecurityManager` con un metodo generale `checkPermission` (e da `AccessController`).

### 14.6 ProtectionDomain

Un **ProtectionDomain** viene creato da un `CodeSource` e una `PermissionCollection` (insieme di permessi), e definisce l'insieme di permessi concessi a una classe. Ogni classe appartiene a un'unica istanza di `ProtectionDomain`, impostata al momento della creazione e **non modificabile** successivamente. L'accesso a questi oggetti è ristretto. [p. 79]

### 14.7 Policy Class

La **Policy Class** è l'interfaccia per definire politiche di sicurezza particolari. Dato un `CodeSource`, restituisce una `PermissionCollection`. Viene invocata durante il caricamento di una classe (dal ClassLoader) per impostare il `ProtectionDomain` appropriato. [p. 80]

### 14.8 Impostazione del ProtectionDomain

Il flusso di impostazione del `ProtectionDomain` coinvolge i seguenti attori: [p. 81]

1. Il **ClassLoader** carica le classi (`Class1`, `Class2`, …).
2. Per ciascuna classe, il ClassLoader interroga l'istanza di **Policy**.
3. La Policy restituisce il `ProtectionDomain` appropriato (contenente `CodeSource` e `PermissionCollection`).
4. Ogni istanza della classe viene associata al proprio `ProtectionDomain`.

### 14.9 Caricamento di una classe in Java 2

Quando una classe `C1` richiede di caricare (*load*) una classe `C2` non ancora caricata, il processo è il seguente: [p. 82]

> [!info] Flusso di caricamento di una classe
> 1. Il **ClassLoader** di `C1` cerca il file `C2.class`, lo carica e chiama il **Bytecode Verifier**.
> 2. Il **CodeSource** di `C2` viene determinato (URL di provenienza e firma).
> 3. L'oggetto **Policy** del ClassLoader restituisce le **Permissions** dato il CodeSource di `C2`.
> 4. Se un `ProtectionDomain` già esistente ha lo stesso `CodeSource` e le stesse `Permissions`, viene **riusato**; altrimenti viene **creato un nuovo** `ProtectionDomain`.
> 5. `C2` viene assegnata al suo `ProtectionDomain`.
> 6. La classe può essere istanziata e i suoi metodi eseguiti; i **controlli runtime** continuano durante l'esecuzione.

### 14.10 Controllo run-time della sicurezza

Il meccanismo di controllo a tempo di esecuzione funziona nel modo seguente: [p. 83]

Se una classe `C` invoca un metodo `M` che richiede un permesso `P`:

1. La JVM chiama `checkPermission(P)` del **SecurityManager** corrente.
2. Questo attiva un **AccessController** che controlla il chiamante di `M`, cioè `C`.
3. Se il `ProtectionDomain` di `C` contiene il permesso `P`, l'operazione procede normalmente.
4. In caso contrario, viene sollevata un'**eccezione**.

> [!warning] Catena delle chiamate
> L'AccessController controlla **tutte le chiamate di metodi nella catena delle chiamate** (*call stack inspection*): è sufficiente che una qualsiasi classe nella catena non abbia il permesso richiesto perché l'operazione venga bloccata.
