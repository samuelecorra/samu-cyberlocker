## 2. Criteri di test strutturali — Richiamo

Ricordiamo che un criterio $C : P \times T \to \{\text{True}, \text{False}\}$ può essere inteso anche come **generatore** di test set: $C : P \to T$ tale che $C(P, T) = \text{True}$. [p. 3]

Nel **program-based testing**, il criterio di selezione dei test si deriva dalla struttura stessa del programma (in particolare, dal grafo di flusso). I criteri principali basati sulla copertura sono:

1. **Copertura delle istruzioni** (statement coverage)
2. **Copertura degli archi** (branch coverage)

## 3. Statement Coverage

### 3.1 Definizione

Un test set $T$ è adeguato per testare un programma $P$ secondo il criterio di **copertura delle istruzioni** se, per **ogni istruzione** $s$ di $P$, esiste un caso di test in $T$ che esegue $s$. [p. 4]

> [!example] Esempio
> ```c
> if (x != 0) then x := x + 10 else x := x - 10
> ```
> Per soddisfare la copertura delle istruzioni, $T$ deve contenere almeno due casi: uno con $x = 0$ e uno con $x \neq 0$, ad esempio $T = \{x = 0,\; x = 10\}$.

### 3.2 Esempi di Statement Coverage

> [!example] Esempio 1 — if/else
> ```c
> vs: int main(int x) {
>      int result;
> v1:  if (x > 0)
> v2:      result = x;
>      else
> v3:      result = 1 / x;
> v4:  printf("%d", result);
> vf: }
> ```
> Istruzioni: `vs, v1, v2, v3, v4, vf`. [p. 5]
> - $x = 1$ copre `vs, v1, v2, v4, vf`
> - $x = -1$ copre `vs, v1, v3, v4, vf`
> - Insieme: copertura completa.

### 3.3 Fault Detection Capability dello Statement Coverage

Come si valuta un criterio di copertura? Tramite la **fault detection capability**: quali errori è in grado di trovare e, soprattutto, quali **garantisce** di trovare. [p. 8]

> [!warning] Limiti dello Statement Coverage
> - Le istruzioni **sempre errate** vengono individuate.
> - Gli **errori nelle decisioni** non è detto che vengano trovati.
> - Questo criterio è **debole**, ma potrebbe bastare come *"minimal criterion for commercial software"*.

### 3.4 Misura di copertura delle istruzioni ($C_0$)

Un caso di test (o una test suite) può soddisfare **parzialmente** un criterio. La misura di copertura per lo statement coverage è: [p. 9]

$$C_0 = \frac{\text{numero di statement eseguiti}}{\text{numero di statement eseguibili}}$$

Una test suite $T$ **soddisfa** il criterio di copertura delle istruzioni se $C_0 = 1$.

> [!example] Misura parziale vs completa
> Per il programma `select` dell'Esempio 3 di poco fa: [p. 10]
> - $(N=1, A[0]=7, X=9)$ copre 5/6 istruzioni → $C_0 = 5/6$ (parzialmente adeguato).
> - $(N=1, A[0]=-7, X=9)$ copre 6/6 istruzioni → $C_0 = 1$ (adeguato).

## 4. Branch Coverage (copertura degli archi)

### 4.1 Definizione

Un test set $T$ soddisfa il criterio di **copertura degli archi** (branch coverage) di $P$ se e solo se **ogni arco** (branch) del grafo di $P$ è percorso almeno una volta. [p. 11]

> [!example] Esempio — if senza else
> ```c
> if (x < 0) then x++; endif
> ```
> Per il branch coverage servono almeno due casi: uno con $x < 0$ e uno con $x \geq 0$, ad es. $T = \{x = -3,\; x = 10\}$.

![](../imgs/Pasted%20image%2020260318045427.png)

### 4.2 Branch Coverage — Esempio

Per il programma dell'esempio 3 `select`: [p. 12]

- Servono 2 casi di test: $(N=1, A[0]=-7, X=9)$ e $(N=1, A[0]=7, X=9)$.
- Alternativa: un singolo caso $(N=2, A=[-7, 7], X=9)$ può coprire entrambi i rami.

![](../imgs/Pasted%20image%2020260318045618.png)

### 4.3 Relazione tra Statement e Branch Coverage

> [!info] Relazione fondamentale
> Lo **statement coverage è più debole** del branch coverage. [p. 13]
>
> - Il **branch coverage implica** lo statement coverage: se un test set $T$ soddisfa il branch coverage, soddisfa anche lo statement coverage.
> - Ma **non viceversa**: un test set che soddisfa lo statement coverage non necessariamente soddisfa il branch coverage.

![](../imgs/Pasted%20image%2020260318045654.png)

### 4.4 Misura di copertura degli archi ($C_{path}$)

Un test $T$ può soddisfare il branch coverage **parzialmente**. La misura è: [p. 14]

$$C_{path} = \frac{\text{numero di branch eseguiti}}{\text{numero di branch eseguibili}}$$

Una test suite soddisfa il criterio se $C_{path} = 1$.

> [!example] Misura parziale vs completa
> Per il programma con `if (x > 0)` dell'Esempio 1: [p. 15]
> - $T = \{1\}$ copre 1 arco su 2 → $C_{path} = 50\%$ (parzialmente adeguato).
> - $T = \{1, -1\}$ → $C_{path} = 1$ (soddisfa il branch coverage).

![](../imgs/Pasted%20image%2020260318045749.png)

## 5. Decision Coverage e Condition Coverage

### 5.1 Decisioni e Condizioni — Distinzione

È fondamentale distinguere: [p. 17]

- **Decisione**: predicato (espressione booleana) guardia di una istruzione condizionale (`if`) o iterativa (`while`, `for`…). Esempio: `if (x > 0 || y > 0)`.
- **Condizione**: espressione booleana **atomica** (non divisibile in sotto-espressioni più semplici) che appare in una decisione. Nell'esempio sopra: `x > 0` e `y > 0`.

> [!example] Esempio — Decisione e condizioni
> ```c
> vs: int main(int x, y) {
>        int result;
> v1:    if (x > 0 || y > 0)
> v2:        result = x;
>        else
> v3:        result = y;
> v4:    printf("%d", result);
> vf: }
> ```
> La **decisione** è `(x > 0 || y > 0)`. Le **condizioni** sono `x > 0` e `y > 0`. [p. 18]

![](../imgs/Pasted%20image%2020260318050026.png)

### 5.2 Copertura delle decisioni (Decision Coverage)

Copre **ogni decisione e la sua negazione**. È **equivalente al branch coverage**: ogni arco nel grafo di flusso viene percorso. [p. 19]

Un test set $T$ è adeguato secondo il criterio di copertura delle decisioni se, per **ogni decisione** di $P$, esiste:
- un caso di test in $T$ in cui la decisione è **presa** (true);
- un caso di test in $T$ in cui la decisione **non è presa** (false).

> [!example] Esempio
> Per `if (x > 0 || y > 0)`: [p. 20]
> - $t_1$: $x = 1, y = 1$ → copre la decisione (true)
> - $t_2$: $x = -1, y = -1$ → copre la negazione (false)

![](../imgs/Pasted%20image%2020260318050125.png)

### 5.3 Copertura delle condizioni (Condition Coverage)

Copre **ogni condizione atomica e la sua negazione** all'interno di tutte le decisioni. [p. 21]

$$C_{cond} = \frac{\text{numero di valori di verità assunti da tutte le condizioni}}{\text{numero di condizioni} \times 2}$$

Una test suite $T$ soddisfa il criterio se $C_{cond} = 1$.

> [!example] Esempio — Test Set 1 (copre condizioni E decisioni)
> Per `if (x > 0 || y > 0)`: [p. 22]
> - $t_1$: $x = 1, y = 1$ → copre `x > 0` (T), `y > 0` (T) → decisione vera
> - $t_2$: $x = -1, y = -1$ → copre `!(x > 0)` (F), `!(y > 0)` (F) → decisione falsa
>
> Questo test set soddisfa **sia** la copertura delle condizioni **sia** la copertura delle decisioni.

![](../imgs/Pasted%20image%2020260318050503.png)

### 5.4 Incomparabilità tra Decision e Condition Coverage

> [!warning] Osservazione A — La copertura delle condizioni NON implica la copertura delle decisioni
> Test Set 2 per `if (x > 0 || y > 0)`: [p. 23]
> - $t_1$: $x = -1, y = 1$ → copre `!(x > 0)` (F), `y > 0` (T) → decisione **vera**
> - $t_2$: $x = 1, y = -1$ → copre `x > 0` (T), `!(y > 0)` (F) → decisione **vera**
>
> Tutte le condizioni sono coperte (T e F per ciascuna), **ma la decisione è sempre vera** → la copertura delle decisioni **non è soddisfatta**.

![](../imgs/Pasted%20image%2020260318050517.png)

> [!warning] Osservazione B — La copertura delle decisioni NON implica la copertura delle condizioni
> Test Set 3 per `if (x > 0 || y > 0)`: [p. 24]
> - $t_1$: $x = 1, y = -1$ → decisione vera
> - $t_2$: $x = -1, y = -1$ → decisione falsa
>
> Le decisioni sono coperte, **ma** `y` non è mai $> 0$ → la copertura delle condizioni **non è soddisfatta**.

![](../imgs/Pasted%20image%2020260318050535.png)

> [!abstract] Conclusione
> I criteri di copertura delle **condizioni** e delle **decisioni** sono **incomparabili** (A + B). [p. 24]

### 5.5 Copertura delle Decisioni e delle Condizioni (combinato)

Il criterio combinato richiede la copertura sia di **tutte le decisioni** che di **tutte le condizioni**: [p. 25]

> [!example] Esempio
> Per `if (x > 0 || y > 0)`:
> - $t_1$: $x = 1, y = 1$ → decisione vera + condizioni `x > 0` (T), `y > 0` (T)
> - $t_2$: $x = -1, y = -1$ → decisione falsa + condizioni `!(x > 0)` (F), `!(y > 0)` (F)

![](../imgs/Pasted%20image%2020260318050610.png)

### 5.6 Short-circuit evaluation

I compilatori spesso usano la **valutazione a corto circuito** per efficienza: [p. 26]

- `a && b`: se `a` è falso, **non valuta** `b`.
- `a || b`: se `a` è vero, **non valuta** `b`.

> [!tip] Nota
> Esistono operatori che evitano il corto circuito (es. `&` e `|` al posto di `&&` e `||`). Il corto circuito **complica** la scelta dei casi di test.

> [!example] Esempio con short-circuit
> Per la decisione `(y > 0 || x > 0)`: [p. 27]
>
> | Test | `y > 0` | `x > 0` | Decisione |
> |---|---|---|---|
> | $y = 1, x = 1$ | T | **non valutata** | T |
> | $y = 0, x = 1$ | F | T | T |
> | $y = 0, x = 0$ | F | F | F |
>
> Per testare `x > 0`, la prima condizione **deve** essere falsa, altrimenti `x > 0` non verrebbe valutata.

> **Fine del Blocco 2** — [pp. 16–27] ✅

## 6. Riepilogo criteri visti finora

Abbiamo visto i criteri di copertura: [p. 28]

1. Delle **istruzioni** (statement coverage) — tutte le istruzioni vengono eseguite.
2. Dei **branch** — tutti gli archi nel grafo di flusso vengono percorsi.
3. Delle **decisioni** — equivalente al branch coverage.
4. Delle **condizioni** — ogni condizione atomica assume T e F.
5. Delle **decisioni/condizioni** — combinazione dei due.

Ora vediamo come coprire **ulteriormente** le condizioni all'interno delle decisioni:
6. **Multiple Condition Coverage** (MCC)
7. **Modified Condition/Decision Coverage** (MCDC)

---

## 7. Multiple Condition Coverage (MCC)

### 7.1 Definizione

Un test set soddisfa **MCC** se testa **ogni possibile combinazione** dei valori di verità delle condizioni atomiche in ogni decisione. [p. 29]

Con $n$ condizioni in una decisione si avranno $2^n$ combinazioni (T e F per ogni condizione). Si rappresenta in genere con una **tabella**. Il numero si riduce grazie alla **short-circuit evaluation**.

> [!example] Esempio — MCC per `(digit_high == -1 || digit_low == -1)`
> [p. 30]
>
> | Test Case | `digit_high == -1` | `digit_low == -1` | Decisione |
> |---|---|---|---|
> | 1 | T | (non valutato) | T |
> | 2 | F | T | T |
> | 3 | F | F | F |

> [!warning] Problema dell'MCC
> Il numero di test cresce **esponenzialmente** con il numero $n$ di condizioni. Se $n$ è grande, l'MCC **non è fattibile**. [p. 30]

## 8. Modified Condition/Decision Coverage (MCDC)

### 8.1 Motivazione

Per superare la crescita esponenziale dell'MCC, sono stati introdotti criteri più forti del semplice condition coverage ma **lineari** con $n$. L'**MCDC** ha avuto particolare successo ed è **richiesto dalla FAA** per il software degli aeromobili commerciali. [p. 31]

> [!quote] Riferimento
> Chilenski, J., Miller, S. *"Applicability of modified condition/decision coverage to software testing"*. Software Engineering Journal, September 1994, pp. 193–200.

### 8.2 Definizione

> [!info] MCDC — Definizione
> Il test deve mostrare che ogni **condizione atomica** influenzi l'esito della decisione **in modo indipendente** dalla valutazione delle altre condizioni. [p. 32]
>
> Per ogni condizione atomica $C$, esistono **due casi di test** in cui:
> - I valori di verità di **tutte le condizioni eccetto** $C$ sono gli stessi.
> - La **decisione** è valutata True per un caso e False per l'altro.

Il test set deve essere scelto in modo che ogni condizione all'interno di una decisione faccia variare **in modo indipendente** il valore finale della decisione.

### 8.3 Esempio 1 — AND e OR

Per ogni condizione $C$ in una decisione $D$: varia solo $C$, le altre restano uguali, e il valore di $D$ cambia. [p. 33]

**Caso $D = A \;\text{and}\; B$:**
- Varia $A$ (tengo $B$ vero): $[A=\text{T}, B=\text{T}]$ (Dec=T) vs $[A=\text{F}, B=\text{T}]$ (Dec=F)
- Varia $B$ (tengo $A$ vero): $[A=\text{T}, B=\text{T}]$ (Dec=T) vs $[A=\text{T}, B=\text{F}]$ (Dec=F)
- Bastano **3 casi**: $\{[A,B],\; [\lnot A, B],\; [A, \lnot B]\}$

**Caso $D = A \;\text{or}\; B$:**
- Varia $A$: $[A=\text{T}, B=\text{F}]$ (Dec=T) vs $[A=\text{F}, B=\text{F}]$ (Dec=F)
- Varia $B$: $[A=\text{F}, B=\text{T}]$ (Dec=T) vs $[A=\text{F}, B=\text{F}]$ (Dec=F)
- Bastano **3 casi**: $\{[\lnot A, \lnot B],\; [A, \lnot B],\; [\lnot A, B]\}$

### 8.4 Esempio 2 — Decisione composta

Per la decisione: [pp. 34–35]

```
if (Reset == on) and (Pressure == TooLow or Pressure == Normal) then ...
```

Con 3 condizioni atomiche: `Reset == on`, `Pressure == TooLow`, `Pressure == Normal`.

| `Reset == on` | `Pressure == TooLow` | `Pressure == Normal` | Decisione |
|---|---|---|---|
| T | T | – | T |
| T | F | T | T |
| T | F | F | F |
| F | T | – | F |

Bastano **4 casi di test** (anziché $2^3 = 8$ dell'MCC). [p. 35]

> **Fine del Blocco 3** — [pp. 28–40] ✅

## 9. Path Testing (Test dei cammini)

### 9.1 Definizione

Il **path testing** è una strategia il cui obiettivo è eseguire ogni **percorso indipendente** di un programma: [p. 41]

- Tutte le istruzioni vengono eseguite almeno una volta.
- Tutte le decisioni vengono testate a true e false.
- I casi di test vanno scelti in modo che **ciascun percorso** del grafo di flusso sia attraversato almeno una volta.

> [!info] Relazione con il branch testing
> Il path testing **implica** sempre il branch testing, ma il branch testing **non implica** il path testing. Nel branch coverage un singolo test case copre solo un arco; nel path testing si coprono interi cammini. [p. 41]

### 9.2 Complessità ciclomatica (CC)

Il **numero minimo** di test necessari per la copertura di tutti i path è pari al numero di **path indipendenti** del grafo di flusso, definito come: [p. 42]

$$CC = E - N + 2$$

dove:
- $E$ = numero di **archi** (transizioni) nel grafo
- $N$ = numero di **nodi** (decisioni + azioni + inizio + fine)

### 9.3 Esempio — Path testing con if annidati

```c
f(x, y) =
    if (x > 0) then
        if (y > 0) then print("A")
        else print("B")
    else print("C")
```

Con $N = 7$ nodi, $E = 8$ archi → $CC = 8 - 7 + 2 = 3$. [p. 43]

$T_P = \{(1, 1),\; (1, -1),\; (-1, \_)\}$ copre il path testing con 3 casi di test.

Per il branch coverage basterebbero in principio 2 casi di test per decisione, ma i 3 casi di $T_P$ li coprono comunque.

### 9.4 Esempio — `getNumDaysInMonth()`

Il grafo di flusso del metodo `getNumDaysInMonth()` ha una complessità ciclomatica $CC = 6$, che corrisponde esattamente al numero di casi di test trovati nella test suite per il path coverage. [pp. 44–46]

### 9.5 Esempio — `isLeapYear()`

Il metodo `isLeapYear()` ha complessità ciclomatica $CC = 2$. [p. 47]

### 9.6 Esercizio — Branch ⊄ Path

> [!example] Esercizio
> ```c
> esempio(a, b) =
>     if (a > 0) then
>         if (b > 0) then print("A")
>     if (b > 0) then print("B")
> ```
> Questa è una funzione per cui un test set adeguato per la **copertura dei rami** non riesce a coprire i **cammini**. [p. 48]

> **Fine del Blocco 4** — [pp. 41–48] ✅

## 10. Equivalence Testing

### 10.1 Ipotesi di base

L'**ipotesi del test di equivalenza** afferma che i sistemi di solito si comportano in modo **simile** per tutti i membri di una classe. Per testare il comportamento di una classe di equivalenza, è sufficiente testarne **solo un membro**. [p. 49]

### 10.2 Fasi dell'equivalence testing

Il test di equivalenza si articola in **due fasi**: [p. 49]

1. **Identificazione delle classi di equivalenza**:
   - I possibili input sono partizionati in classi di equivalenza (set di input che verranno **trattati allo stesso modo** nel codice).
   - Le partizioni devono includere non solo gli input corretti, ma anche le **"partizioni di non correttezza"** con input deliberatamente errati.

2. **Selezione** di un test case per ogni classe.

### 10.3 Criteri per le classi di equivalenza

I seguenti criteri sono utilizzati per determinare le classi: [p. 51]

> [!info] Tre proprietà delle classi di equivalenza
> - **Copertura**: ogni possibile input appartiene a **una** delle classi.
> - **Disarticolazione**: nessun input appartiene a **più di una** classe.
> - **Rappresentazione**: se si osserva un malfunzionamento con un membro di una classe, lo stesso malfunzionamento si osserva con **qualsiasi altro** membro della stessa classe.

### 10.4 Esempio — `calcola_prezzo(eta)`

```python
calcola_prezzo(eta):
    if eta < 0 or eta > 120: return "Errore"
    elif eta < 12:            return "Biglietto bambino"
    elif eta < 65:            return "Biglietto adulto"
    else:                     return "Biglietto anziano"
```

Il dominio valido è $0 \leq \text{eta} \leq 120$. Le classi di equivalenza: [p. 52]

| Classe | Intervallo | Valore scelto | Output atteso |
|---|---|---|---|
| Non valida | $\text{eta} < 0$ | $-5$ | "Errore" |
| Valida | $0 \leq \text{eta} < 12$ | $5$ | "Biglietto bambino" |
| Valida | $12 \leq \text{eta} < 65$ | $30$ | "Biglietto adulto" |
| Valida | $65 \leq \text{eta} \leq 120$ | $80$ | "Biglietto anziano" |
| Non valida | $\text{eta} > 120$ | $150$ | "Errore" |

### 10.5 Esempio — `getNumDaysInMonth(month, year)`

Per `getNumDaysInMonth(int month, int year)`: [p. 53]

- **3 classi di equivalenza per il mese**: mesi da 31 gg, mesi da 30 gg, febbraio (< 30 gg).
- **2 classi di equivalenza per l'anno**: bisestili e non bisestili.
- **Input non validi**: interi negativi per mese e anno, interi > 12 per mese.

## 11. Boundary Testing

Piuttosto che selezionare un elemento qualsiasi nella classe di equivalenza, il **boundary testing** richiede che gli elementi siano selezionati **al bordo** della classe (valori limite). [p. 54]

> [!tip] Valori aggiunti dal boundary testing
> - Febbraio come valore limite per i falsi bisestili.
> - Valori limite nella rappresentazione del mese (0, 1, 12, 13…).

## 12. State-based Testing

### 12.1 Definizione

Nel contesto di una classe, lo **state-based testing** (o state transition testing) deriva i casi di test da una **macchina a stati** (UML) per la classe. [p. 57]

- **Test case**: sequenza di stimoli o eventi per testare la copertura di path di esecuzione (sequenze di transizioni), la raggiungibilità di stati specifici, la raggiungibilità di tutti gli stati finali.
- **Test set**: insieme di casi di test adeguati per raggiungere la copertura richiesta.

### 12.2 Esempio — Macchina a stati di un orologio

La macchina ha gli stati **MeasureTime**, **SetTime** e **DeadBattery**. Le transizioni sono: [pp. 58–59]

| # | Stato sorgente | Evento / Azione | Stato destinazione |
|---|---|---|---|
| 1 | (iniziale) | — | MeasureTime |
| 2 | MeasureTime | pressLeftButton / pressRightButton | MeasureTime |
| 3 | MeasureTime | pressBothButtons | SetTime |
| 4 | SetTime | after 2 min. | MeasureTime |
| 5 | MeasureTime | pressBothButtons / beep | MeasureTime |
| 6 | SetTime | pressLeftButton / pressRightButton | SetTime |
| 7 | MeasureTime | batteryEmpty | DeadBattery |
| 8 | SetTime | batteryEmpty | DeadBattery |

### 12.3 Test case e copertura delle transizioni

Ogni singolo test case copre una o più transizioni: [p. 60]

| Test Case | Sequenza di input | Copre transizioni |
|---|---|---|
| TC1 | Avvio sistema | 1 |
| TC2 | pressLeftButton | 2 |
| TC3 | pressBothButtons | 3 |
